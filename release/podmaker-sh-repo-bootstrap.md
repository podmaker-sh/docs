# podmaker-sh organisation — repo bootstrap

Topology: **Pattern C — 4 repo.**

| Repo                            | Visibility | Purpose                                                  |
|---------------------------------|------------|----------------------------------------------------------|
| `podmaker-sh/podmaker`          | private    | Monorepo (apps/* + packages/* + control plane)           |
| `podmaker-sh/releases`          | public     | Binary release host — CI cross-repo pushes here          |
| `podmaker-sh/homebrew-tap`      | public     | Brew tap — `Formula/podmaker-vault-bridge.rb` lives here |
| `podmaker-sh/docs`              | public     | Customer-facing documentation                            |

## One-time setup commands

Run as `podmaker-bot` (machine user) or org admin.

```sh
# 1. Private monorepo (push current cwd here)
gh repo create podmaker-sh/podmaker --private \
    --description "PodMaker monorepo — control plane, agents, providers"
cd ~/Sites/hostingmanager   # legacy cwd path; renaming optional
git remote add origin https://github.com/podmaker-sh/podmaker.git
git push -u origin main

# 2. Public release host (no code, just receives tagged release assets)
gh repo create podmaker-sh/releases --public \
    --description "PodMaker release artefacts (binaries, MSI, .deb, .rpm, image attestations)"
gh repo edit podmaker-sh/releases \
    --enable-issues=false --enable-projects=false --enable-wiki=false

# 3. Public homebrew tap
gh repo create podmaker-sh/homebrew-tap --public \
    --description "PodMaker Homebrew tap — brew install podmaker-sh/tap/<formula>"
cd /tmp && git clone https://github.com/podmaker-sh/homebrew-tap.git && cd homebrew-tap
mkdir -p Formula && touch Formula/.gitkeep
cat <<TXT > README.md
# PodMaker Homebrew tap

\`\`\`sh
brew tap podmaker-sh/tap
brew install podmaker-vault-bridge
\`\`\`

Formula files are auto-generated by the \`release-vault-bridge\`
workflow on every \`bridge-v*\` tag in the private monorepo.
TXT
git add . && git commit -m "Bootstrap" && git push

# 4. Public docs (customer-facing surface)
gh repo create podmaker-sh/docs --public \
    --description "PodMaker public documentation"
cd /tmp && git clone https://github.com/podmaker-sh/docs.git && cd docs
mkdir -p setup release runbooks
cat <<TXT > README.md
# PodMaker docs

Customer-facing documentation for vault-bridge agent, installer
scripts, runbooks. Synced periodically from the monorepo
docs/ tree by the CI publish-docs workflow.
TXT
git add . && git commit -m "Bootstrap" && git push
```

## Container registry

```sh
# Ghcr.io packages live under the org. Push happens automatically
# from the release workflow; only manual step is making the image
# public on the first push:
#
#   GitHub → podmaker-sh org → Packages → vault-bridge-agent
#   → Package settings → Change visibility → Public
#
# Or via API once published:
gh api -X PATCH /orgs/podmaker-sh/packages/container/vault-bridge-agent \
    -f visibility=public
```

## Required secrets on `podmaker-sh/podmaker`

| Secret                           | Purpose                                       |
|----------------------------------|-----------------------------------------------|
| `RELEASES_PAT`                   | Cross-repo push to `podmaker-sh/releases`     |
| `HOMEBREW_TAP_PAT`               | Cross-repo push to `podmaker-sh/homebrew-tap` |
| `DOCS_PAT`                       | Cross-repo push to `podmaker-sh/docs` (optional) |
| `WINDOWS_CODESIGN_CERT_B64`      | EV cert PFX (base64) — MSI signing            |
| `WINDOWS_CODESIGN_CERT_PASS`     | PFX export password                           |

PAT scopes: each one is a fine-grained token issued by
`podmaker-bot`, scoped to the single target repo with `Contents:
read+write`. Rotate annually.

```sh
gh secret set RELEASES_PAT      --repo podmaker-sh/podmaker --body "<paste>"
gh secret set HOMEBREW_TAP_PAT  --repo podmaker-sh/podmaker --body "<paste>"
gh secret set DOCS_PAT          --repo podmaker-sh/podmaker --body "<paste>"
```

## Verifying the first release

```sh
# From the monorepo:
git tag bridge-v0.1.0
git push origin bridge-v0.1.0
```

Expected behaviour:

1. `binaries` job: cross-compiles → uploads to
   `podmaker-sh/releases` release `bridge-v0.1.0`.
2. `packages` job: builds .deb + .rpm → uploads to same release.
3. `msi` job: builds + signs MSI → uploads to same release.
4. `image` job: pushes multi-arch image to
   `ghcr.io/podmaker-sh/vault-bridge-agent:v0.1.0`.
5. `brew-tap` job: rewrites
   `podmaker-sh/homebrew-tap/Formula/podmaker-vault-bridge.rb`.

Smoke test from a customer machine:

```sh
curl -fsSL https://app.podmaker.sh/install/vault-bridge | sh
brew tap podmaker-sh/tap && brew install podmaker-vault-bridge
docker pull ghcr.io/podmaker-sh/vault-bridge-agent:latest
```

## Public docs sync

The monorepo's `docs/` tree is the source of truth. A nightly
workflow rsyncs the customer-facing subset to `podmaker-sh/docs`:

```
docs/setup/*       → docs repo /setup/
docs/release/*     → docs repo /release/
docs/runbooks/*    → docs repo /runbooks/
```

Internal-only files (`docs/architecture/`, `docs/sprint-plans/`)
stay private.

This sync workflow lives at
`.github/workflows/publish-docs.yml` and runs daily at 02:30 UTC.
