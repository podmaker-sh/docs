# Homebrew tap — manual bootstrap

The `brew-tap` job inside `release-vault-bridge.yml` pushes formula
updates to a SEPARATE repo named `homebrew-tap`. That repo does NOT
exist by default; create it once before running the workflow.

## One-time setup

1. **Create the tap repository.**

   ```bash
   gh repo create podmaker/homebrew-tap --public --description "PodMaker Homebrew tap"
   ```

   The repo MUST live under the same org as the main podmaker repo
   so the workflow's `${{ github.repository_owner }}` resolves
   correctly. Public is fine — formulas advertise the same release
   URLs everyone can already hit on GitHub.

2. **Seed the directory layout.**

   ```bash
   git clone https://github.com/podmaker/homebrew-tap.git
   cd homebrew-tap
   mkdir -p Formula
   touch Formula/.gitkeep
   cat <<'TXT' > README.md
   # PodMaker Homebrew tap

   Run `brew tap podmaker/tap` and then `brew install podmaker-vault-bridge`.
   Formula files are auto-generated by the `release-vault-bridge`
   workflow on every `bridge-v*` tag.
   TXT
   git add . && git commit -m "Bootstrap" && git push
   ```

3. **Provision the `HOMEBREW_TAP_PAT` secret.** See
   `docs/release/vault-bridge-release-secrets.md` for the exact
   scope. The release workflow pushes commits as `podmaker-bot`
   via this PAT.

4. **Smoke test the workflow.**

   ```bash
   gh workflow run release-vault-bridge.yml -f version=v0.0.0-tap-bootstrap
   ```

   When the `brew-tap` job completes, `Formula/podmaker-vault-bridge.rb`
   appears in the tap with sha256 placeholders replaced by real values.

## Verifying a customer install

```bash
brew tap podmaker/tap
brew install podmaker-vault-bridge
podmaker-vault-bridge --help

# Or run as a background service via brew services.
brew services start podmaker-vault-bridge
```

The first run drops an mTLS cert under `~/.podmaker-bridge/` and
prints the install snippet from the CP — that snippet contains the
bridge id + bearer token the operator already pasted into the
service env file (`/usr/local/etc/podmaker-vault-bridge.env` or
the `EnvironmentFile` referenced by the brew formula service
stanza).

## Rolling the beta channel

Beta builds publish to `Formula/podmaker-vault-bridge-beta.rb`
(separate formula, same tap). Customers opt in via:

```bash
brew install podmaker-vault-bridge-beta
```

The CI rewriter knows which formula to update based on whether the
tag ends in `-beta.N`.

## Nightlies

Skipped on purpose. Nightlies move too fast for Homebrew's policy
of stable binaries; the install script handles them via `?channel=nightly`.
